## Solo Brainstorm — angles non couverts par le brainstorm joint

**Faisabilité de l'extraction (Phase A)**
- Le digest (`transcript-digest.ts`) préserve la séquence des tours, donc l'extraction des problèmes et schémas est faisable. MAIS budget 9000 chars head+tail → milieu tronqué sur sessions longues. **Décision** : augmenter le budget pour le pipeline enrichi (les problèmes peuvent vivre au milieu). Risque résiduel : sessions très longues restent partielles → acceptable v1.
- `turns_to_resolve / backtracks / tool_errors` par problème = **estimations LLM** (le LLM lit la séquence), pas des comptages déterministes. Suffisant pour amorcer la sélection. Durcissement possible plus tard (mesure transcript). Noté.

**Edge cases**
- Session sans problème (tout fluide) → `problems: []`, difficulty `easy`. Fitness N/A.
- Problème non résolu (`outcome: unresolved`) → exclu de l'élection de champion (un schéma qui n'a pas marché n'est pas un champion) mais conservé pour le recurrence tracking.
- Catégorie de problème vue une seule fois → champion trivial (lui-même), pas de comparaison. Devient signifiant à 2+.
- Schéma v1 (sessions déjà consolidées sans `problems`) → rétro-compat : champs absents = `[]`/defaults. Bump `SUMMARY_SCHEMA_VERSION` 1→2 ; les anciens résumés restent lisibles, re-consolidables à la demande.

**Dépendances implicites**
- Nouveaux fichiers d'état : `champions.json`, `evolution.json`, `injections.json` (dans `stateDir()/`).
- Hooks Claude Code (`SessionStart`, `UserPromptSubmit`) → config `~/.claude/settings.json` (hors repo) + scripts dans le repo. **Anti-récursion** : respecter `ARCADE_LOOP_ACTIVE=1` (déjà en place) pour ne pas s'auto-déclencher pendant la conso.
- Coût tokens : ré-consolidation enrichie des sessions = dépense cloud Sonnet. Le backfill complet reste gaté (déclenchement manuel).

**Risques d'intégration**
- `SummaryFields`/`SessionSummary` sont partagés (parse, store, insights, graph, api, web). Changement de type = ripple. → Story de types partagés en premier, additive uniquement (champs optionnels au narrowing).
- `rebuildInsights()` dans `run.ts` est le point de reconstruction déterministe → y brancher champions + evolution (cheap, zéro LLM).

**Contraintes d'ordre**
- A.1 (types) avant tout. Puis prompt/parse/digest en parallèle. Puis tests.
- B (champions) dépend de A. C (evolution) dépend de B. D (push) dépend de B. E (app) dépend de B/C/D.
